## 为什么 TDD 与 AI 编程是天作之合

传统上，TDD 很难坚持——写测试比写功能枯燥，而且需要对需求有清晰的理解才能写出好测试。

Claude Code 改变了这个局面：

- **AI 帮你写测试** — 从需求描述直接生成全面的测试用例，包括边界情况
- **测试约束代码行为** — 先有测试，Claude 生成的代码有明确目标
- **即时反馈** — 让 Claude 运行测试，看到红/绿状态立即修正

结果：**更少的 back-and-forth，更高的代码质量**。

## Red-Green-Refactor 循环

TDD 的核心是三步循环：

```
🔴 RED：先写一个失败的测试
         ↓
🟢 GREEN：写最少的代码使测试通过
         ↓
🔵 REFACTOR：优化代码，保持测试通过
         ↓
（回到 RED，继续下一个功能点）
```

每个循环应该**很小**——通常只测试一个行为点。

## 让 Claude 先写测试

对 Claude Code 来说，"先写测试"意味着：

```
1. 描述功能需求（而不是直接要求实现）

> 我需要实现一个 slugify 函数：
  - 将字符串转为 URL 友好格式
  - 大写转小写
  - 空格和特殊字符替换为连字符
  - 移除首尾连字符
  - 连续连字符合并为一个

  先为这个函数写完整的测试用例（Jest/Vitest），
  包括：正常情况、边界值（空字符串、全特殊字符）、中文输入、
  长字符串、已经是 slug 格式的字符串

  测试写好后，不要写实现，先让我看测试
```

这样得到的测试会覆盖你可能没想到的边界情况。

## 从测试失败到实现通过

测试确认后，再要求实现：

```
2. 测试确认后

> 好，这些测试覆盖得很全。现在运行测试，
  确认它们全部失败（因为函数还不存在）

3. 运行测试，看到全红

> 现在实现 slugify 函数，目标是让所有测试通过。
  用最简单的实现，不需要额外的功能

4. 运行测试，看到全绿

> 所有测试通过了。现在看看实现是否有可以重构的地方？
  （但要保持测试全部通过）
```

## 边界条件与异常路径测试

AI 的一大优势是能系统性地枚举边界条件，不遗漏：

```
> 为 divide(a, b) 函数生成测试，
  要求覆盖所有边界情况：
  - 正常除法
  - 除数为零
  - 负数
  - 浮点数精度
  - 极大值/极小值（Infinity、Number.MAX_SAFE_INTEGER）
  - 非数字输入（null、undefined、字符串）
```

## 为遗留代码补写测试

已有代码没有测试是很常见的情况。让 Claude 反向生成测试：

```
> 读取 UserService.java（约 150 行），
  分析所有执行路径（正常流程、条件分支、异常处理），
  生成覆盖率目标 90% 的单元测试套件（JUnit 5 + Mockito）

  重点覆盖：
  - 每个 public 方法的正常情况
  - if/else 的每个分支
  - 可能抛出异常的地方
  - 边界输入（null、空集合、超长字符串）
```

## 实战：TDD 开发一个工具函数库

```
步骤 1：
> 我要开发一个日期工具库（dateUtils.js），
  包含以下函数：
  - formatDate(date, format) — 格式化日期
  - isWeekend(date) — 判断是否周末
  - addDays(date, n) — 加减天数
  - diffDays(date1, date2) — 计算相差天数

  先为这四个函数生成完整测试（Vitest），
  不写任何实现代码

步骤 2：
> 运行测试，确认全部失败

步骤 3：
> 逐个实现函数，每实现一个就运行测试，
  看到绿色后再继续下一个

步骤 4：
> 全部测试通过后，检查实现代码是否有重构空间
```

title: TDD 实现字符串工具库
difficulty: 中级

用 TDD 方式开发一个字符串工具库，包含三个函数：

1. `slugify(str)` — 转 URL 友好格式（含中文支持）
2. `truncate(str, maxLen)` — 截断字符串，超长加 `...`
3. `escapeHtml(str)` — 转义 HTML 特殊字符（`<>&"'`）

**严格遵守顺序**：先写所有测试 → 确认全红 → 再实现 → 确认全绿

---hints---
- 让 Claude 一次为三个函数写所有测试，然后你审查确认
- 确认测试合理后，说「现在运行测试，应该全部失败」
- 再说「现在逐个实现这三个函数，每完成一个就运行测试」
- 中文 slugify 可以选择直接移除中文，或者用拼音转换（简单起见移除即可）

---solution---
示例对话流程：

```
> 用 TDD 方式开发 stringUtils.js，
  先为 slugify、truncate、escapeHtml 三个函数写完整的 Vitest 测试。
  测试要覆盖正常情况、边界值（空字符串、null）、特殊字符。
  暂时不写实现。

[Claude 生成测试文件]

> 先运行这些测试，确认都失败（因为函数还没实现）

> 好，现在按顺序实现：先实现 slugify，运行测试看是否通过，
  通过后再实现 truncate，最后实现 escapeHtml

> 全部通过后，检查代码质量，有没有可以改进的地方？
```
