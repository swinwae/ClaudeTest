## 单 Agent 的局限性

一个对话中的单个 Claude 实例：
- 只能串行处理，一次做一件事
- 上下文有限，无法同时深入多个模块
- 单一视角，容易有盲点
- 长任务会让上下文窗口超限

多 Agent 编排打破这些限制。

## 多 Agent 编排模式

### 模式一：扇出-汇总（Fan-out / Fan-in）

```
                 ┌─ Agent 1：安全分析 ─┐
主 Agent ──────── ├─ Agent 2：性能分析 ─┤──── 主 Agent 汇总
                 └─ Agent 3：质量审查 ─┘
```

适合：同一个对象需要从多个维度独立分析。

```
> 对 src/auth/ 目录同时启动三个分析任务：
  1. 安全漏洞扫描（重点：认证绕过、注入攻击）
  2. 性能瓶颈分析（重点：数据库查询、缓存使用）
  3. 代码可维护性审查（重点：耦合度、测试覆盖）

  三个任务并行执行，完成后我来汇总关键发现
```

### 模式二：流水线（Pipeline）

```
需求分析 → 架构设计 → 测试编写 → 代码实现 → 代码审查
```

每步的输出是下一步的输入。适合有明确依赖关系的复杂流程。

```
步骤 1：> 分析需求：[需求描述]，输出用户故事和验收条件

步骤 2：> 基于以上需求，设计实现方案（不写代码），
         输出：接口设计、数据模型、关键决策

步骤 3：> 基于方案，先写测试（TDD 流程）

步骤 4：> 实现代码使所有测试通过

步骤 5：> 审查实现代码，重点检查安全性和性能
```

### 模式三：分裂角色（Split Role）

对同一个问题，让 Claude 扮演不同的专家角色：

```
> 对以下架构方案，请分别从三个角度提供意见：
  1. 作为安全专家：这个设计有哪些安全风险？
  2. 作为 DBA：数据库层面有哪些问题？
  3. 作为团队 Lead：可维护性如何？新成员上手难度？

  三个角色的分析要分开，最后综合评估
```

## 模型混用策略

不同的任务不需要用同一个模型：

| 任务 | 推荐模型 | 原因 |
|------|---------|------|
| 架构决策、复杂分析 | Opus 4.6 | 最强推理能力 |
| 主要开发工作 | Sonnet 4.6 | 最佳代码质量 |
| 批量轻量任务（日志分析、格式化） | Haiku 4.5 | 成本低 3 倍 |

在对话中切换模型：

```
/model claude-opus-4-6
> 帮我设计这个分布式缓存系统的架构方案

/model claude-sonnet-4-6
> 基于以上架构，实现 CacheManager 类

/model claude-haiku-4-5-20251001
> 为以下 50 个函数生成简短注释（批量任务）
```

## 实战：多 Agent 协作完成功能开发全流程

一个从需求到上线的 5 步 Agent 协作流程：

```
# 阶段 1：规划（使用 planner agent）
> 用 planner agent 分析以下需求，制定实施方案：
  需求：为博客添加文章收藏功能（登录用户可收藏/取消收藏，查看收藏列表）

# 阶段 2：TDD（使用 tdd-guide agent）
> 规划方案已确认，用 tdd-guide agent 先写测试：
  - 收藏/取消收藏的 Service 层测试
  - 收藏列表查询的 Repository 测试

# 阶段 3：实现
> 测试已写好，实现代码使所有测试通过

# 阶段 4：并行审查（同时启动两个任务）
> 启动两个并行审查任务：
  任务 A：code-reviewer agent 审查代码质量
  任务 B：security-reviewer agent 检查安全问题
  两个任务完成后汇总结果

# 阶段 5：修复审查发现的问题，提交 PR
```

---exercise---
title: 三视角并行代码审查
difficulty: 高级
---
选择一个你项目中的核心模块（至少 100 行），用多 Agent 从三个视角同时审查：

1. **安全视角**：有哪些安全漏洞或风险？
2. **性能视角**：有哪些性能瓶颈或低效操作？
3. **可维护性视角**：代码耦合度如何？新人上手难度？

将三份报告汇总，按优先级制定改进计划（CRITICAL/HIGH/MEDIUM/LOW）。

---hints---
- 可以直接让 Claude 模拟"分别扮演三个专家视角"，这不需要真正启动独立进程
- 如果使用 ECC 的 Agent 工具，可以用 `/agent code-reviewer`、`/agent security-reviewer` 等
- 汇总时说：「综合三个视角的报告，按优先级列出需要改进的点」

---solution---
示例对话：

```
> 读取 myblog/src/main/java/com/myblog/service/PostServiceImpl.java

> 现在从三个独立视角分析这个文件：

  视角 1（安全专家）：识别所有潜在的安全问题
  视角 2（性能专家）：识别所有性能瓶颈和低效操作
  视角 3（架构专家）：评估代码的职责划分、耦合度和可测试性

  每个视角独立分析，不要相互影响，最后汇总到一个优先级矩阵

> 根据上面的分析，生成一个 IMPROVEMENT_PLAN.md，
  按 CRITICAL/HIGH/MEDIUM/LOW 排列所有改进项，
  每项包含：问题描述、受影响文件、建议修复方式
```
